### AMB-001-01: `user` 与 `fnd_user_info` 双用户表并存
- **位置**: spec.md 数据契约"表 user"小节;spec-draft.md 数据契约"表 user"小节;FR-022(操作日志 LEFT JOIN `user`)、FR-024(委派查询外键→user.username)、FR-010(密码更新)
- **现象**: 代码中同时存在 `fnd_user_info`(主流程用)与 `user`(日志查询 sql-map-admin-config.xml:563、委派查询 :651-660、MD5 密码更新 :245-247 等多处 LEFT JOIN)两张用户表。spec 将 `user` 标记为 D(废弃),但又在多个 FR 中实际依赖它。两表关系未明确:是同义视图、双写同步、还是历史遗留待迁移。
- **候选解释**:
  1. `user` 是 `fnd_user_info` 的别名/同义视图,指向同一物理表,SQL 中混用仅为历史命名习惯。
  2. 两表为独立物理表,存在数据双写或迁移未完成,查询结果可能不一致。
  3. `user` 是完全废弃的遗留表,代码中引用属未清理的技术债,实际数据已全量迁移至 `fnd_user_info`。
- **影响面**: FR-010(密码重置)、FR-022(日志查询补全 realName)、FR-024(委派查询 owner/assignee);数据契约 user 表分级;SC-007(日志审计真实姓名补全)。
- **建议决策**: 倾向解释 1(同义视图/别名)。理由:日志查询、委派查询等只读场景与主流程写操作若指向不同物理表会导致密码更新后日志仍查老数据,生产环境不可接受。建议新系统统一为单一用户表,迁移期若需兼容则用视图别名过渡。需在 clarify 阶段通过 DDL 确认。

### AMB-001-02: `role` 与 `fnd_roles` 双角色表并存
- **位置**: spec.md 数据契约"表 role"小节;spec-draft.md 数据契约"表 role"小节;FR-005(角色 ID 串转角色名展示,sql-map-admin-config.xml:243-247,261-266,535)
- **现象**: 角色管理主流程使用 `fnd_roles`(FR-012~FR-014),但用户列表查询中 roleIds 转角色名展示、以及部分角色查询仍使用老表 `role`。spec 将 `role` 标记为 D(废弃),但 FR-005 实际依赖它完成角色名展示。两表字段对应关系(id/roleName/status)一致但未确认是否同源。
- **候选解释**:
  1. `role` 是 `fnd_roles` 的别名/视图,同 AMB-001-01。
  2. 两表独立,`role` 为历史遗留,角色名展示查询的是过期数据,存在展示与实际授权不一致风险。
  3. `role` 仅用于只读查询场景(列表展示),写操作走 `fnd_roles`,过渡期靠触发器或同步任务保持一致。
- **影响面**: FR-005(用户列表角色名展示)、FR-007(编辑用户角色关联);数据契约 role 表分级;SC-003(权限校验中角色判定)。
- **建议决策**: 倾向解释 1(别名/视图),与 AMB-001-01 保持一致策略。理由:若展示用的角色名与实际授权角色来自不同表,用户看到的角色名可能与实际权限不符,属安全隐患。建议新系统统一为 `fnd_roles`,迁移期用视图兼容。

### AMB-001-03: 区域权限市场/用服映射规则启用状态不明
- **位置**: spec.md FR-001 步骤6("[待澄清]")、SC-014("[待澄清:映射规则当前是否启用]");spec-draft.md FR-001 步骤6、NFR-017;Assumptions"区域权限映射规则待确认"
- **现象**: FR-001 与 SC-014 描述登录时 areapower 合并规则:若 areapower=`-1` 且 dpNo 非空则替换、若不含 dpNo 则追加,之后再按"市场/用服办事处映射规则补充(16x↔31x 前缀互换)"。但 spec-draft.md NFR-017 注明该 16x↔31x 互换逻辑在 `LoginServiceImpl.processAreaPower` 中"已注释,但最终调用 UserUtil.processAreaPower",实际是否执行未确认。
- **候选解释**:
  1. 映射规则已启用:`LoginServiceImpl` 中虽注释了内联逻辑,但调用链最终经 `UserUtil.processAreaPower` 执行了等价规则。
  2. 映射规则已禁用:注释即表示废弃,`UserUtil.processAreaPower` 可能是空实现或已被移除,16x↔31x 互换不再生效。
  3. 映射规则部分启用:仅特定部门前缀触发,条件需查 `UserUtil` 实现。
- **影响面**: FR-001(登录区域权限计算)、SC-014(区域权限合并验收)、用户登录后可见的数据范围(areapower 决定部门级数据权限)。
- **建议决策**: 需查 `UserUtil.processAreaPower` 源码确认。若无法确认,建议按解释 2(已禁用)处理——新系统不实现 16x↔31x 互换,仅保留 dpNo 合并逻辑;若业务反馈区域权限缺失再补回。理由:注释通常表示废弃意图,新系统不应继承未确认的隐式规则。

### AMB-001-04: 用户角色变更联动(changeType)规则与失败处理缺失
- **位置**: spec.md FR-007 步骤4("[待澄清:具体联动规则]")、SC-013("[待澄清:具体联动规则与失败处理]");spec-draft.md FR-007 步骤4、NFR-016;Assumptions"用户角色变更联动为外部调用"
- **现象**: 编辑用户时若提供 `changeType`(service/program/both)与 `newMemberCode`,需"联动批量更新项目服务经理/项目经理"。spec 注明此联动依赖外部项目工具调用(`ProjectUtils.updateServiceAndProgramMember`),但具体联动规则(哪些项目、更新哪个字段、批量范围)与失败处理(外部调用失败时是否回滚用户编辑、是否重试)完全缺失。
- **候选解释**:
  1. changeType=service 更新所有该用户任服务经理的项目,将服务经理字段替换为 newMemberCode;program 同理对项目经理;both 两者都更新。失败时不回滚用户编辑,仅记录日志。
  2. 同上,但失败时整体回滚(用户编辑与项目更新同事务或补偿)。
  3. 联动为异步消息,用户编辑立即返回,项目更新后续异步执行,失败重试。
- **影响面**: FR-007(编辑用户)、SC-013(角色变更联动验收);用户离职/转岗后项目归属正确性;跨域依赖(项目域)。
- **建议决策**: 倾向解释 1(失败不回滚,仅日志)。理由:联动为外部项目工具调用,跨系统事务成本高且 spec 已标注"外部调用";用户编辑是本域核心职责不应被外部系统故障阻断。新系统建议改为异步消息(解释 3)更健壮。需在 clarify 阶段确认 changeType 三值与项目字段的精确映射。

### AMB-001-05: 角色默认页面硬编码 `module/Welcome1.action`
- **位置**: spec.md FR-013 步骤2("[待澄清:是否应可配置]")、数据契约 fnd_roles.defaultPage;spec-draft.md FR-013 步骤2、附录歧义点3;Assumptions"角色默认页面硬编码(待 clarify)"
- **现象**: 新增/编辑角色时 defaultPage 强制写入固定值 `module/Welcome1.action`(RoleManageAction.java:61,113),代码注释标明"最好不要写死,后期需要调整"。spec 将此标记为待澄清,未决定是保留硬编码还是改为可配置。
- **候选解释**:
  1. 保留硬编码:所有角色统一落地页,用户级 defaultPage 才是个性化的,角色级 defaultPage 实际未被使用。

  3. 角色级 defaultPage 已废弃:实际登录跳转只用用户级 defaultPage(fnd_user_info.defaultPage),角色级字段为历史遗留。

- **建议决策**: 倾向解释 2(改为可配置)。理由:代码注释已表明开发意图为去硬编码;角色级默认页在多角色场景下可作为角色切换的落地页。新系统应将 defaultPage 作为角色表单的可选输入,留空时回退至系统默认页。若确认解释 3(角色级未使用),则应标记为 D 级字段。

### AMB-001-06: `menuCode` 唯一性约束未确认
- **位置**: spec.md 数据契约 fnd_menus 表 menuCode 字段("[待澄清:未见显式 unique,但作为关联键]");spec-draft.md 数据契约 fnd_menus 表
- **现象**: `fnd_user_menus.menuCode` 作为外键关联 `fnd_menus.menuCode`,即 menuCode 是用户菜单授权的关联键。但 spec 注明"未见显式 unique 约束",仅作为关联键推断其应唯一。若 menuCode 不唯一,fnd_user_menus 关联将产生一对多歧义。
- **候选解释**:
  1. menuCode 在数据库层面有唯一索引但未在 SQL-map 中体现(DDL 层约束)。
  2. menuCode 无唯一约束,依赖应用层保证不重复,存在数据质量风险。
  3. menuCode 可重复(如不同层级菜单复用编码),fnd_user_menus 关联时通过 menuCode+其他字段消歧。
- **影响面**: 数据契约 fnd_menus.menuCode、fnd_user_menus.menuCode;FR-015(菜单管理)、FR-001 步骤5(登录加载用户菜单权限 menuCode→menuValue);菜单授权一致性。
- **建议决策**: 倾向解释 1(有唯一约束,DDL 层)。理由:menuCode 作为跨表关联键若无唯一性保证,用户菜单授权查询将不可预测,生产系统不可接受。建议新系统在 menuCode 上显式建唯一索引。若 DDL 确认无约束,则需排查是否存在重复数据并补建。

### AMB-001-07: 测试环境免密登录语义歧义
- **位置**: spec.md FR-001 步骤3、US-001 场景5;spec-draft.md FR-001 步骤3;Assumptions"环境开关语义"
- **现象**: FR-001 步骤3 描述:"若环境参数非 1(测试环境),忽略提交密码,直接使用库中密码"。US-001 场景5描述:"测试环境下,用户提交任意用户名(忽略密码),系统使用库中密码直接放行"。两处表述存在歧义:"忽略提交密码,直接使用库中密码"是指任意密码都能登录(只要用户名存在),还是指系统跳过密码比对但仍要求用户名有效?US-001 场景5说"任意用户名"但"使用库中密码放行"——若用户名不存在何来"库中密码"?
- **候选解释**:
  1. 测试环境:用户名必须存在且有效(status=1),密码任意(或为空)即可登录,系统跳过密码比对。
  2. 测试环境:用户名任意(含不存在),系统自动放行,无需任何有效凭证(完全免登录)。
  3. 测试环境:用户名必须存在,系统用库中密码替换提交密码后比对(等价于密码无关但用户必须存在)。
- **影响面**: FR-001(账号密码登录)、US-001 场景5、Edge Case"用户不存在";测试环境安全策略;SC-004(密码安全验收,测试环境豁免范围)。
- **建议决策**: 倾向解释 1(用户名必须存在有效,密码任意)。理由:FR-001 步骤2仍执行"按用户名查询有效用户",步骤3仅跳过步骤4的密码比对;"使用库中密码"语义为系统内部替换而非用户提交。解释 2(完全免登录)存在安全隐患,不应为生产可部署系统的测试模式。建议 clarify 时确认 `sys.envirment.argu` 的完整值域与非生产值的行为。

### AMB-001-08: CAS 服务器地址硬编码与可开关配置矛盾
- **位置**: spec.md FR-004 步骤1(CAS 登出地址 `https://cas.dptech.com:8443/logout?...`)、NFR-010/SC-008(CAS 可开关);spec-draft.md FR-004、附录歧义点6
- **现象**: NFR-010 声明 CAS 单点登录通过系统参数 `sys.cas` 可开关,但 FR-004 中 CAS 登出地址 `https://cas.dptech.com:8443/` 为硬编码,未参数化。spec-draft.md 附录明确列为歧义点"是否应改为系统参数"。这导致 CAS 开关可控制是否启用,但 CAS 服务器地址无法通过配置变更,换环境需改代码。

### AMB-001-09: `menuPower` 权限编码语义反直觉
- **位置**: spec.md FR-001 步骤7、数据契约 fnd_role_menus.menuPower;spec-draft.md FR-001 步骤7、NFR 引用 LoginServiceImpl.java:63-77
- **现象**: menuPower 权限编码定义为 `8`=新增、`1`=删除、`4`=查询、`2`=更新。此编码与常规权限位模型不一致(常规:1=读/查、2=写/增、4=改、8=删,或 1=create、2=read、4=update、8=delete)。spec 中"8=新增、1=删除"的语义反直觉,且 spec-draft.md 标注证据为 LoginServiceImpl.java:63-77 但未附原文,无法确认是代码实际值还是反推笔误。
- **候选解释**:
  1. 编码正确:代码中确实 `8`=新增、`1`=删除、`4`=查询、`2`=更新,为项目自定义编码体系(非标准位运算)。
  2. 编码标注有误:实际应为 `1`=查询、`2`=新增、`4`=更新、`8`=删除(标准位运算),spec 反推时将值与操作错配。
  3. menuPower 存储的是位运算累加值(如 `1,2,4,8` 组合),单个值含义需看 LoginServiceImpl 解析逻辑而非字面映射。
- **影响面**: FR-001 步骤7(角色菜单操作权限加载)、数据契约 fnd_role_menus.menuPower 值域;前端按钮级权限控制(增/删/查/改按钮显隐);US-12(角色菜单权限分配)。
- **建议决策**: 需查 LoginServiceImpl.java:63-77 源码确认。若确认解释 1(自定义编码),新系统建议标准化为位运算(解释 2 的映射)以降低认知成本;若确认解释 2(spec 笔误),应修正 spec。在 clarify 前以代码源值为准。

### AMB-001-10: 老架构/新架构行为并存于同一 FR,契约边界模糊
- **位置**: spec.md FR-006 步骤6、FR-007 步骤5、FR-009、FR-005"新架构补充";Assumptions"两套端点并存"
- **现象**: 多个 FR 内同时描述老架构(Struts `.action`)与新架构(REST `/system`)两套行为,且行为不同:FR-006 步骤6 老架构写用户菜单授权、新架构写用户角色关联(不同实体);FR-007 步骤5 老架构全删重插菜单授权、新架构增量增删角色关联(不同策略);FR-009 老架构统计计数、新架构按公司判断可用性(不同逻辑)。spec 声明"新系统应统一为一套",但未明确哪些行为是契约(C 级)、哪些是实现细节可丢弃。
- **候选解释**:
  1. 新架构行为为契约:老架构行为标记为过渡实现,新系统以新架构(多租户、compId、角色关联)为准。
  2. 两套行为均为契约:新系统需同时支持用户菜单授权(直接授权)与用户角色关联(角色授权)两种授权模型。
  3. 需逐 FR 区分:部分 FR 新架构为契约(如 FR-008 删除用户的多租户逻辑),部分 FR 老架构仍为契约(如 FR-001 登录流程)。
- **影响面**: FR-006~FR-009(用户 CRUD)、FR-016/FR-017(授权绑定);数据契约(用户菜单授权 vs 用户角色关联两套模型);新系统授权模型设计。
- **建议决策**: 倾向解释 2(两套均为契约,新系统同时支持)。理由:用户菜单授权(绕过角色的直接授权)与用户角色关联(通过角色的间接授权)是两种互补的权限模型,业务上均有需求;FR-001 登录时同时加载用户菜单权限(menuCode→menuValue)和角色菜单权限,说明两者并存是设计而非遗留。建议新系统保留双模型,但统一端点为 REST 风格。

### AMB-001-11: 邮件发送失败重试上限与补偿机制缺失
- **位置**: spec.md SC-009、US-17、Edge Case"邮件发送失败/延迟";spec-draft.md NFR-011;Assumptions"邮件发送为异步队列模式...无显式重试次数上限(待 clarify)"
- **现象**: SC-009 规定邮件按期望发送时间轮询发送,发送后 sendFlag=1。Edge Case 描述"发送失败保留未发标志,后续轮询重试"。但 spec 未定义:重试次数上限、重试间隔、达到上限后的处理(死信队列/告警/丢弃)、邮件服务器不可用时的降级策略。Assumptions 明确标注"无显式重试次数上限(待 clarify)"。
- **候选解释**:
  1. 无限重试:sendFlag 始终为 0,每次轮询都尝试发送,直到成功或邮件被手动清理。
  2. 固定次数重试:达到上限(如 3 次)后标记为失败(sendFlag=2),不再重试,需人工介入。
  3. 指数退避重试:重试间隔递增,达到上限后进入死信状态。
- **影响面**: SC-009(邮件可靠性验收)、US-17(邮件通知);fnd_mails.sendFlag 值域(当前仅 {0,1},是否需扩展失败态);账号开通/密码重置通知的可到达性。
- **建议决策**: 倾向解释 2(固定次数 + 失败标记)。理由:无限重试(解释 1)在邮件服务器持续不可用时会积压大量请求;新系统应扩展 sendFlag 值域(0=未发、1=已发、2=失败),达到重试上限后标记失败并告警。建议 clarify 时确认老代码是否有重试计数字段(如 retryCount)或仅靠 sendFlag 判断。

### AMB-001-12: 多租户 `compId` 隔离在老架构缺失导致双语义
- **位置**: spec.md FR-008(删除用户按 compId)、FR-009(新架构按公司判断可用性)、FR-017(用户角色绑定补 compId)、NFR-007/SC-005;spec-draft.md NFR-007;Assumptions"多租户为新架构能力"
- **现象**: NFR-007 声明用户扩展信息(UserInfo)与用户角色关联(UserRole)均按 compId 隔离,系统用户 compId=-1(跨租户)。但老架构(Struts)无 compId 概念,fnd_user_info 表无 compId 字段(数据契约中未列出),用户与角色的关联通过 fnd_user_info.roleIds(单字段 `;id;` 串)实现而非独立的 UserRole 关联表。spec 同时描述了两套用户-角色关联模型:老架构(roleIds 串)与新架构(UserRole 表 + compId),未明确新系统采用哪套。
- **候选解释**:
  1. 新系统采用 UserRole 表 + compId 模型:roleIds 串标记为废弃,迁移时将 roleIds 拆解为 UserRole 记录。
  2. 新系统保留 roleIds 串 + 引入 compId 字段:用户主表增加 compId,角色关联仍用 roleIds 串但按 compId 隔离。
  3. 双模型并存:roleIds 串用于老架构兼容(无多租户),UserRole 表用于新架构(多租户),迁移期共存。
- **影响面**: FR-006~FR-009(用户 CRUD)、FR-016/FR-017(角色绑定);数据契约 fnd_user_info.roleIds vs UserRole 表;NFR-007(多租户隔离);SC-005(跨租户越权拦截);用户-角色关联的查询/增删逻辑。
- **建议决策**: 倾向解释 1(UserRole 表 + compId)。理由:NFR-007 已明确多租户隔离为新架构能力,roleIds 串无法表达"同一用户在不同公司有不同角色"的多租户语义;UserRole 表(userId+roleId+compId)是表达多租户角色关联的标准模型。建议新系统采用 UserRole 表,roleIds 串仅作迁移期只读兼容,迁移完成后标记为 D 级。需 clarify 时确认 UserInfo 扩展信息表的字段结构(spec 中未给出完整 DDL)。
