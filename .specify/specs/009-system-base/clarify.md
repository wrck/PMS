## AMB-009-01: 系统变量加密算法术语不一致(ASE vs AES)

- **决策结论**: 统一采用 AES 对称加密;草稿中的 `ASE` 视为 `AES` 的笔误。spec 中 SC-007、表 `t_sys_variable` 的 `var` 字段说明与业务规则均补充为"AES 加密"。
- **依据**: `ASE` 非业界标准对称加密算法名,且与 AES 仅一字之差,笔误可能性最高;统一术语可消除迁移时加密互操作风险。
- **影响范围**: SC-007;表 `t_sys_variable`(var 字段语义、业务规则);新系统迁移时的加密实现。
- **回滚提示**: 若源码 `SystemVariableController.java:60-66` 确认实际为项目自定义算法(非 AES),需将 spec 中 AES 改回自定义算法全名,并补全加解密实现说明。

## AMB-009-02: 系统变量时效过滤条件边界不一致(`<=` vs `<`)

- **决策结论**: 各实体按其既定规则过滤——新架构统一 `effectiveFrom < now()`;老架构系统变量(`fnd_sys_arg`)保留 `effectiveFrom <= now()`(含生效时刻)。SC-024 表述修正为"各实体按其既定规则"。
- **依据**: 老架构 SQL `effectiveFrom <= now()` 为历史实现,强行统一可能改变老系统行为;新架构以 `<` 为准保持一致性。
- **影响范围**: SC-024;FR-2.7.1(老架构系统变量查询);表 `fnd_basic_data`、`t_notify_template` 时效规则。
- **回滚提示**: 若新系统迁移要求全量统一边界,需在迁移阶段统一为 `<=` 或 `<`,并同步修正 SC-024 与各实体查询 SQL。

## AMB-009-03: 邮件失败重试边界表述歧义("超过" vs "达到")

- **决策结论**: 以代码 `failedCount < maxCount` 为准,SC-025 修正为"达到 `maxCount` 后不再重试",明确"最多重试 maxCount 次(失败计数从 0 开始)"。Edge Cases 标题由"超过上限"改为"达到上限"。
- **依据**: 代码查询条件 `failedCount < 3` 拾取意味着 `failedCount=0,1,2` 重试、`=3` 跳过,即"达到即跳过";SC-025 原"超过"表述与代码行为差 1。
- **影响范围**: SC-025;Edge Cases(邮件失败);测试用例若以 SC-025 为基准需同步修正。
- **回滚提示**: 若确认代码存在 off-by-one 应为 `<=`,需修正代码而非 spec,并同步更新 SC-025 边界。

## AMB-009-04: 受限 SQL 执行功能(FR-2.1.5)是否保留

- **决策结论**: [暂定决策] 新系统废弃裸 SQL 执行入口(FR-2.1.5),紧急运维改走 SC-006 三层校验受控通道;老系统保留并标记为弃用,迁移时移除。US-1 场景 5 与 FR-2.1.5 风险行的 `[待澄清]` 已替换为决策。
- **依据**: FR-2.1.5 仅校验 `where`/`insert` 关键字,与 SC-006 三层校验安全模型冲突,保留裸 SQL 入口将形成安全旁路;废弃后紧急运维可由受控通道承接。
- **影响范围**: FR-2.1.5;US-1 场景 5;Edge Cases(无 WHERE 的 UPDATE/DELETE);SC-006 安全模型一致性;管理员运维流程。
- **回滚提示**: 若运维强烈要求保留裸 SQL 入口,须强制接入 SC-006 三层校验(HTML 净化 + 注入正则 + 表白名单)后再放行,并补充管理员二次审批。

## AMB-009-05: 公开文件下载端点鉴权策略未定(SC-013)

- **决策结论**: [暂定决策] 公开下载端点须校验 `t_file.downloadKey`(无 downloadKey 或不匹配则拒绝),防止 ID 枚举遍历。SC-013、FR-2.5.7、US-5 场景 4 的 `[待澄清]`/待确认已替换。

- **影响范围**: SC-013;FR-2.5.7;US-5 场景 4;表 `t_file`(downloadKey 字段);SC-016 下载日志可追溯性。
- **回滚提示**: 若源码 `UploaderController.java:258-288` 的 `publicDownload` 确认完全无鉴权,需评估是否加时效签名替代 downloadKey,或接受公开无鉴权并放弃 SC-016 对公开下载的追溯。

## AMB-009-06: 新老架构表是否同步保持一致(SC-022)

- **决策结论**: [暂定决策] 过渡期新老表分别服务不同模块(老模块读写老表、新模块读写新表),无双向同步;迁移时按子域分批合并到新表,合并前老表只读保留。SC-022 与 Edge Cases 的 `[待澄清]` 已替换。
- **依据**: 四组新老表字段命名/类型差异大,双向同步成本高且易不一致;分别服务不同模块可在过渡期保持各自一致性,分批合并降低风险。
- **影响范围**: SC-022;Edge Cases(新老架构表并存);Assumptions(过渡期并存);`fnd_basic_data`/`t_dictionary`、`fnd_mails`/`t_mails`、`fnd_sys_arg`/`t_sys_variable`、`tb_sys_log`/`t_sys_log` 四组表;DATA-REUSE-01 契约。
- **回滚提示**: 若确认存在双写或定时同步任务,需在 spec 中为每组表补充"写入方/读取方/同步策略"三栏,并明确过渡期数据一致性容忍度。

## AMB-009-10: 报表临时表并发冲突与 DROP 时机未明确

## AMB-009-11: 印章导入 truncate 与 INSERT 事务性未明确

- **决策结论**: [暂定决策] 新系统改用事务内 `DELETE` + INSERT 替代非事务的 `truncate`,失败可回滚;导入前可选备份。FR-2.3.3 增加事务策略行,Edge Cases 修正。
- **依据**: `truncate` 是 DDL(MySQL 非事务不可回滚),INSERT 失败将导致 `mes_seal_info` 清空且不可恢复,合格证查询(FR-2.3.2)全部失效;事务内 DELETE 可回滚保障数据安全。
- **影响范围**: FR-2.3.3;Edge Cases(印章导入覆盖);表 `mes_seal_info` 业务规则;FR-2.3.2 合格证查询可用性。
- **回滚提示**: 若 `mes_seal_info` 数据量极大导致 DELETE 性能不可接受,可改为"先备份再 truncate"或"INSERT 到临时表成功后 RENAME 切换"。

## AMB-009-17: t_sys_log.create_date 为 VARCHAR 类型与时间查询适配未说明

- **决策结论**: [暂定决策] 新系统新增 TIMESTAMP 列为 C 级,原 VARCHAR `create_date` 降为 I 级保留;字符串格式约定 `yyyy-MM-dd HH:mm:ss`。表 `t_sys_log` 已修正字段分级与格式说明。
