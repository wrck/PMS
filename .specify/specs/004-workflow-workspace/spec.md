# Feature Specification: 004-workflow-workspace(工作流与工作空间)

**Feature Branch**: `004-workflow-workspace`

**Created**: 2026-07-09

**Status**: Draft

**Source**: 逆向反推自 PMS-struts + PMS-activiti

## User Scenarios & Testing

> 域职责:BPMN 流程引擎管理、任务管理、委派、工作空间(待办聚合)。
> 角色体系:流程管理员、系统管理员、办理人、回访角色(ROLE_CALLBACKPER)、技术公告角色(ROLE_PROB_ADMIN/SUPPORTER/RD)、区域负责人(ROLE_AREA_LEADER)、财务(ROLE_FINANCIAL_STAFF)、工程经理。

### User Story 1 - 办理人待办任务处理 (Priority: P1)

作为办理人,我希望查看分配给我的待办任务列表、打开对应业务表单、填写审批意见并提交,使流程流转到下一节点。这是工作流系统的核心日常操作闭环,覆盖待办查询→表单打开→任务提交三个环节。

**Why this priority**: 待办任务处理是所有流程办理人的高频日常操作,直接影响审批效率与业务流转。若仅实现此故事,系统已可作为任务办理终端交付价值。

**Independent Test**: 可通过"以办理人身份登录→查看待办列表→打开一条任务表单→填写意见并提交→验证任务从待办列表消失且流转到下一办理人"完整测试,交付独立的任务办理能力。

**Acceptance Scenarios**:

1. **Given** 办理人已登录且存在分配给该办理人的运行中任务,**When** 办理人打开待办任务列表,**Then** 系统按当前用户名作为办理人查询运行中任务,返回该办理人名下的待办任务列表(含项目编码、项目名、流程类型等描述信息),支持分页。
2. **Given** 办理人待办列表中存在一条任务,**When** 办理人点击该任务打开表单,**Then** 系统按任务ID取表单元数据(formKey),从流程实例业务键(格式 `{classType}.{objId}`)解析业务对象ID,拼装并跳转到表单 URL(含 objId、taskId、canSee、procType 参数)。
3. **Given** 办理人已打开任务表单并填写审批结果与意见,**When** 办理人提交任务,**Then** 系统设置当前认证用户,按业务键解析 classType 与 objId 执行业务侧状态更新,完成任务并写入流程变量(outcome、issamecustomer、needleader),重定向到工作台业务办理页。
4. **Given** 提交任务时业务键格式不符合 `{classType}.{objId}` 约定,**When** 办理人提交任务,**Then** 业务对象解析失败,流程无法正确定位业务数据,任务提交异常。
5. **Given** 驳回场景提交任务,**When** 办理人选择驳回结果并提交,**Then** 系统应根据 outcome 与 classType 执行业务侧状态更新(驳回分支当前存在注释代码,实际生效情况见 FR-TASK-03)。

### User Story 2 - 工作台待办聚合视图 (Priority: P1)

作为登录用户,我希望进入工作台后默认看到日常项目跟踪待办,并能切换到"业务流程办理"标签查看各类业务审批待办(闭环流程、回访、项目回退、不予跟踪、项目督查、售前),以便聚焦当前需要处理的事项。

**Why this priority**: 工作台是用户登录后的首要入口,聚合展示待办是工作流系统面向办理人的核心价值。默认视图与业务流程办理标签覆盖绝大多数日常审批场景。

**Independent Test**: 可通过"登录→查看默认日常项目跟踪待办→切换到业务流程办理标签→验证各类业务待办按流程Key聚合展示"完整测试,交付独立的工作台待办聚合能力。

**Acceptance Scenarios**:

1. **Given** 用户已登录,**When** 用户进入工作台默认视图,**Then** 系统根据 tabIndex 派发,默认查询项目经理待办任务(项目状态在 30/31/32、有效、计划偏离 7 天内、项目经理非空),支持按项目名、办事处、项目经理、服务经理筛选与分页。
2. **Given** 用户具备相应角色且在工作台,**When** 用户切换到"业务流程办理"标签,**Then** 系统按 procKey 聚合多类待办(闭环流程、回访、项目回退、不予跟踪、项目督查[仅工程经理角色]、售前),返回聚合后的任务描述列表;procKey 为空时查询全部。
3. **Given** 用户角色受限(如无回访角色),**When** 系统执行标签裁剪,**Then** 仅具备角色对应的标签可见(回访角色才显示"已办";技术公告角色仅显示技术公告;区域/财务角色才显示转包)。

### User Story 3 - 流程部署管理 (Priority: P2)

作为流程管理员,我希望查看已部署的流程包与流程定义列表、上传 BPMN 压缩包发布新流程、删除废弃部署、查看流程图与当前节点坐标,以便管理系统中可用的流程。

**Why this priority**: 流程部署管理是管理员维护操作,非办理人日常高频操作,但对系统流程的可用性与可维护性至关重要。

**Independent Test**: 可通过"以管理员身份登录→查看部署列表→上传一个 BPMN zip 包→验证新流程定义出现→查看流程图→删除部署→验证流程定义消失"完整测试,交付独立的流程部署管理能力。

**Acceptance Scenarios**:

1. **Given** 管理员已登录,**When** 管理员进入部署管理页,**Then** 系统并行查询所有部署记录与所有流程定义记录(均按引擎默认排序),返回部署列表(部署ID、名称、部署时间)与流程定义列表(定义ID、Key、名称、版本、部署ID、资源名、流程图资源名)。
2. **Given** 管理员选择了 BPMN 压缩包(zip)文件,**When** 管理员执行发布,**Then** 系统将 zip 包作为部署资源、以文件名作为部署显示名执行部署,成功后重定向回部署管理页。
3. **Given** 管理员选择了格式非法或未找到的部署文件,**When** 管理员执行发布,**Then** 系统打印堆栈并设置错误信息,返回错误页。
4. **Given** 管理员指定了部署ID,**When** 管理员执行删除,**Then** 系统以非级联模式删除该部署(不删除相关运行时实例),成功后重定向回部署管理页。
5. **Given** 管理员/办理人指定了部署ID与图片资源名,**When** 请求查看静态流程图,**Then** 系统按部署ID+资源名读取字节流并输出 PNG 图片。
6. **Given** 办理人指定了任务ID,**When** 请求查看当前节点高亮流程图,**Then** 系统按任务ID查流程定义,再查当前活动节点坐标(x、y、width、height),返回流程定义对象与坐标信息。

### User Story 4 - 任务历史回溯 (Priority: P2)

作为办理人,我希望查看已办理任务的历史表单与"已办理"历史任务列表,以便回溯审批记录。

**Why this priority**: 历史回溯是审批合规性与可追溯性的重要保障,但非实时办理的核心路径,优先级低于待办处理。

**Independent Test**: 可通过"以办理人身份登录→提交一条任务→查看已办理历史→打开历史任务表单→验证审批记录可回溯"完整测试,交付独立的历史回溯能力。

**Acceptance Scenarios**:

1. **Given** 办理人指定了流程实例ID,**When** 办理人查看历史任务表单,**Then** 系统按实例ID查历史业务对象ID,按实例ID取首条任务的 formKey,拼装并跳转 URL(含 objId、instId、flag=1、showflag=1 参数)。
2. **Given** 办理人已登录且具备回访角色,**When** 办理人切换到"已办理"标签,**Then** 系统查询当前用户已办理任务(项目回访评价 + 回访流程已办),支持按项目名、办事处、客户筛选与分页。

### User Story 5 - 系统通知与已读 (Priority: P2)

作为用户,我希望查看系统通知消息并能标记已读,以便及时了解项目相关信息。

**Why this priority**: 通知机制提升信息传达效率,但依赖通知产生方,非工作流核心闭环,优先级中等。

**Independent Test**: 可通过"登录→查看未读通知→标记一条已读→验证已读列表更新"完整测试,交付独立的通知查看与已读能力。

**Acceptance Scenarios**:

1. **Given** 用户已登录且存在未读通知,**When** 用户查看通知列表,**Then** 系统返回当前用户通知(未读按创建时间倒序 + 已读按查看时间倒序限 100 条)。
2. **Given** 用户查看了一条未读通知,**When** 系统更新通知状态为已读,**Then** 通知状态置为已读(查看时间置为当前时间)。注:通知状态更新接口在代码中标注为已废弃。

### User Story 6 - 角色专属待办标签 (Priority: P3)

作为技术支持人员,我希望查看技术公告待办;作为区域负责人/财务人员,我希望查看项目转包待办。

**Why this priority**: 角色专属待办面向特定角色群体,受众范围较窄,非通用功能,优先级较低。

**Independent Test**: 可通过"以技术支持角色登录→查看技术公告待办→以区域/财务角色登录→查看转包待办"分别测试,交付独立的角色专属待办能力。

**Acceptance Scenarios**:

1. **Given** 用户具备技术公告角色(管理员/支持/研发),**When** 用户切换到技术公告标签,**Then** 系统按角色查询技术公告待办,过滤已关闭/已解决状态,返回技术公告列表。
2. **Given** 用户具备区域负责人或财务角色,**When** 用户切换到项目转包标签,**Then** 系统查询项目转包任务列表,同时加载有效公司列表(status=1),返回转包任务列表与公司列表。

### User Story 7 - 管理员任务管理 (Priority: P3)

作为系统管理员,我希望查看所有在执行阶段的任务,以便全局监控流程执行情况。

**Why this priority**: 管理员任务管理当前为空实现(仅返回成功视图),实际能力是否由前端或其他动作承担不明,功能可用性存疑,优先级最低。

**Independent Test**: 当前仅能验证请求返回成功视图(实际任务管理能力是否完全由前端或其他动作承担见 FR-TASK-05 与 Edge Cases)。

**Acceptance Scenarios**:

1. **Given** 管理员已登录,**When** 管理员请求任务管理页,**Then** 系统返回成功视图(当前为空实现)。[待澄清] 是否依赖前端直接查询待确认。

### User Story 8 - 委派规则管理 (Priority: P3)

作为办理人/管理员,我希望查询、添加、编辑、更新流程定义委派规则,以便在特定时间段内将任务委派给他人办理。

**Why this priority**: 委派功能在服务层与动作层核心逻辑均被注释,实际为废弃功能,不可用。仅保留表结构与查询入口,优先级最低。

**Independent Test**: 仅能验证列表查询返回(按设置人过滤)。新增/编辑/更新核心逻辑被注释,无法独立测试(委派功能废弃详情见 Edge Cases)。

**Acceptance Scenarios**:

1. **Given** 办理人/管理员已登录,**When** 请求委派规则列表,**Then** 系统按 handleUsername(设置人)过滤查询委派规则,关联用户表展示真实姓名。
2. **Given** 用户请求新增/编辑/更新委派规则,**When** 提交操作,**Then** 系统仅返回成功视图(核心逻辑被注释,功能实际不可用)。

### Edge Cases

- **删除部署存在运行中实例**:非级联删除模式不删除运行时实例,可能导致孤儿流程实例。[待澄清] 是否有补偿或限制策略待确认。
- **业务键格式不符**:提交任务时业务键不符合 `{classType}.{objId}` 约定,将导致业务对象解析失败,任务提交异常。该约定为跨域隐式契约,破坏将导致任务提交失败。
- **部署文件格式非法**:上传非 zip 或损坏文件时,系统打印堆栈并返回错误页。
- **驳回业务状态更新失效**:提交任务时驳回场景的业务状态更新分支被大量注释,实际驳回可能不生效(见 FR-TASK-03)。
- **空待办列表**:用户无任何待办任务时,系统应返回空列表而非错误。
- **无对应角色**:用户无回访/技术公告/区域/财务角色时,对应标签不可见。
- **已读通知超过100条**:已读通知按查看时间倒序限 100 条,超出部分不展示。
- **流程定义不存在**:打开任务表单时 formKey 查询失败或流程定义缺失,需有降级处理。
- **多实例节点终止流程**:终止流程时需处理多实例节点完成条件表达式,动态修改流向并还原。
- **认证上下文异常被静默吞没**:启动流程与提交任务时设置认证身份失败被静默处理(不阻断流程,见 SC-008)。
- **委派功能废弃但表与SQL保留**:委派规则表与完整 SQL 存在,但应用层 CRUD 方法全部被注释,功能实际不可用。[待澄清] 需确认是历史遗留还是计划恢复。
- **dp_act_proc_desc 表疑似未落地**:传输对象注释声称对应此表,但全库无任何 SQL 引用,实为纯传输对象(DTO)。[待澄清] 是否曾规划独立任务描述表待确认。
- **管理员任务管理空实现**:管理员任务管理动作方法体为空(仅返回成功),管理员任务管理能力是否由前端或其他动作承担不明。[待澄清] 实际任务管理能力是否完全由前端或其他动作承担待确认。
- **独立流程建模/浏览应用边界不明**:存在独立的流程模型器/浏览器应用(含独立前端技术栈与配置),与主系统共用引擎表但为独立应用。[待澄清] 其与主系统的工作流管理边界、是否在生产部署未明。

## Requirements

### Functional Requirements

> 子域组织:流程部署管理 / 任务办理 / 委派规则管理 / 工作空间 / 审批意见与流程控制。

#### 流程部署管理

- **FR-DEPLOY-01 查看部署列表与流程定义列表**:系统 MUST 并行查询所有部署记录与所有流程定义记录(均按 BPMN 流程引擎默认排序),返回部署列表(部署ID、名称、部署时间)与流程定义列表(定义ID、Key、名称、版本、部署ID、资源名、流程图资源名)。

- **FR-DEPLOY-02 发布流程**:系统 MUST 支持上传 BPMN 压缩包(zip)发布新流程——将 zip 包作为部署资源,以文件名作为部署显示名执行部署;成功后重定向回部署管理页;文件未找到/格式非法时打印堆栈并设置错误信息,返回错误页。

- **FR-DEPLOY-03 删除部署**:系统 MUST 支持按部署ID删除部署——当部署ID非空时,以非级联模式删除该部署(不删除相关运行时实例),成功后重定向回部署管理页。[待澄清] 非级联删除会导致运行中的流程实例成为孤儿,是否有补偿机制待确认。

- **FR-DEPLOY-04 查看流程图**:系统 MUST 支持两种流程图查看——静态图:按部署ID+资源名读取字节流并输出 PNG 图片;当前节点高亮图:按任务ID查流程定义,再查当前活动节点坐标(x、y、width、height),返回流程定义对象与坐标信息。

#### 任务办理

- **FR-TASK-01 查询个人待办任务**:系统 MUST 按当前登录用户名作为办理人查询运行中任务,扩展展示任务描述信息(项目编码、项目名、流程类型等),返回任务描述列表,支持分页。

- **FR-TASK-02 打开任务表单**:系统 MUST 按任务ID取表单元数据(formKey),按任务ID查流程实例业务键并按 "." 拆分出业务对象ID,拼装表单 URL(formKey?param.objId={objId}&param.taskId={taskId}&param.canSee={canSee}&dpActProcDesc.procType={procType})并跳转。业务键格式约定为 `{classType}.{objId}`。

- **FR-TASK-03 提交任务**:系统 MUST 支持办理人提交任务(含审批结果、意见、是否同客户、是否需领导审批等流程变量)——设置当前认证用户;按任务ID查流程实例,从业务键解析 classType 与 objId;根据 outcome 与 classType 执行业务侧状态更新;完成任务并写入流程变量(outcome、issamecustomer、needleader);成功后重定向到工作台业务办理页。[待澄清] 驳回业务状态更新分支当前被大量注释,实际驳回是否生效待确认。

- **FR-TASK-04 查看历史任务表单**:系统 MUST 按流程实例ID查历史业务对象ID,按实例ID取首条任务的 formKey,拼装 URL(formKey?param.objId={objId}&param.instId={instId}&param.flag=1&param.showflag=1)并跳转。

- **FR-TASK-05 管理员任务管理**:系统 MUST 支持管理员请求任务管理页——当前为空实现,仅返回成功视图。[待澄清] 实际任务管理能力是否完全由前端或其他动作承担待确认。

#### 委派规则管理

- **FR-DELEGATE-01 查询委派规则列表**:系统 MUST 按 handleUsername(设置人)过滤查询委派规则,关联用户表展示真实姓名。查询有效规则时需 status=1 且当前时间在 startTime/endTime 区间内。

- **FR-DELEGATE-02 新增委派规则**:系统 MUST 支持新增委派规则——当前动作层核心逻辑被注释,仅返回成功视图,功能实际不可用。

- **FR-DELEGATE-03 编辑委派规则**:系统 MUST 支持编辑委派规则——当前动作层核心逻辑被注释,仅返回成功视图,功能实际不可用。

- **FR-DELEGATE-04 更新委派规则**:系统 MUST 支持更新委派规则——当前服务层方法体被注释(空实现),功能实际不可用。委派功能在应用层整体废弃,数据表与 SQL 仍保留。

#### 工作空间

- **FR-WS-01 工作台默认视图(日常项目跟踪)**:系统 MUST 在用户登录后进入工作台时根据 tabIndex 派发,默认查询项目经理待办任务(项目状态在 30/31/32、有效、计划偏离 7 天内、项目经理非空),支持按项目名、办事处、项目经理、服务经理筛选与分页;prepare() 中根据角色裁剪可见标签(回访角色才显示"已办";技术公告角色只显示技术公告;区域/财务角色显示转包)。

- **FR-WS-02 业务流程办理标签**:系统 MUST 按 procKey(空则查全部)聚合多类待办——闭环流程、回访、项目回退、不予跟踪、项目督查(仅工程经理角色)、售前,返回聚合后的任务描述列表。

- **FR-WS-03 已办理历史标签**:系统 MUST 查询当前用户已办理任务(项目回访评价 + 回访流程已办),支持按项目名、办事处、客户筛选与分页。

- **FR-WS-04 技术公告标签**:系统 MUST 按角色(管理员/支持/研发)查询技术公告待办,过滤已关闭/已解决状态,返回技术公告列表。

- **FR-WS-05 项目转包标签**:系统 MUST 查询项目转包任务列表,同时加载有效公司列表(status=1),返回转包任务列表与公司列表。

- **FR-WS-06 系统通知与已读**:系统 MUST 查询当前用户通知(未读按创建时间倒序 + 已读按查看时间倒序限 100 条),并支持更新通知状态为已读(查看时间置为当前时间)。注:通知状态更新接口在代码中标注为已废弃。

#### 审批意见与流程控制

- **FR-COMMENT-01 自定义审批意见**:系统 MUST 提供独立于 BPMN 流程引擎内置批注的自定义审批意见写入与查询,记录办理人、办理时间、结果、意见、下一办理人。

- **FR-PROC-01 终止流程**:系统 MUST 提供按任务ID终止流程的能力——动态修改流程定义流向,将当前节点连接到结束节点,完成任务后还原流向;支持多实例节点完成条件表达式处理。

## Success Criteria

> 由非功能需求转换为可测量标准。所有标准均可通过业务行为验证,不依赖具体技术实现。

### Measurable Outcomes

- **SC-001 任务列表分页**:日常项目跟踪、通知、已办历史等所有列表 100% 支持分页导航,用户可在任意数据量下按页浏览待办。
- **SC-002 待办查询性能**:待办查询(含多表关联与 UNION ALL)在数据量增长后仍保持可接受响应时间,终端用户无感知延迟。[待澄清] 数据量增长后的性能保障(是否有索引)待确认。
- **SC-003 引擎与业务统一事务**:流程操作与业务状态更新 100% 在同一事务内完成,不存在部分提交导致的数据不一致。
- **SC-004 引擎表结构自动管理**:BPMN 流程引擎表结构由引擎自动创建/升级,业务侧不手动维护其 DDL。
- **SC-005 业务键解析准确率**:流程实例业务键 100% 遵循 `{classType}.{objId}` 格式,提交任务时解析准确率 100%;格式破坏将导致任务提交失败,该约定为跨域隐式契约。
- **SC-006 邮件通知可达**:流程邮件通知通过配置的邮件服务器发送,正常网络条件下送达率 100%。
- **SC-007 统一任务监听器**:任务生命周期事件(create/assignment/complete/delete/激活/挂起)100% 触发自定义监听逻辑(如同步到第三方系统);项目验收监听器挂载于所有用户任务。
- **SC-008 认证身份记录**:启动流程与提交任务前 100% 设置当前认证用户身份,完成后清除。[待澄清] 认证身份设置失败被静默处理(不阻断流程),是否会导致审计缺失待确认。
- **SC-009 工作台标签角色控制**:工作台标签 100% 按角色正确裁剪——回访角色才显示"已办";技术公告角色仅显示技术公告;区域/财务角色才显示转包;无对应角色的标签 0 可见。

## Assumptions

- **用户认证与会话**:用户已通过系统认证并具备有效会话,当前登录用户名可用于待办查询与认证身份设置。
- **BPMN 流程引擎可用**:BPMN 流程引擎已正确配置并与业务数据共用同一数据源与事务管理器,保证流程操作与业务状态更新的事务一致性。
- **流程定义已发布**:流程定义已通过部署管理或独立建模器预先发布,待办查询依赖运行时任务表数据。
- **业务键约定稳定**:流程实例业务键采用 `{classType}.{objId}` 格式,该跨域隐式约定被所有流程遵循,破坏将导致任务提交失败。
- **角色体系已定义**:回访角色、技术公告角色、区域负责人、财务、工程经理等角色及其权限边界由权限域定义,本域据此裁剪工作台标签。
- **各业务表数据完整**:工作空间待办聚合依赖各业务表(回访、售前、项目主表、项目督查等)数据完整性,关联缺失将导致待办展示不全。
- **邮件服务器可达**:配置的邮件服务器可正常访问,用于流程邮件通知发送。
- **委派功能废弃状态**:委派功能虽表结构与 SQL 完整保留,但应用层 CRUD 方法全部注释,假定当前为废弃状态不可用。
- **审批意见独立存储**:自定义审批意见独立于引擎内置批注表存储,假定两者互不干扰。
- **终止流程流向可还原**:终止流程通过动态修改流程定义流向实现,假定完成任务后能可靠还原原始流向。
- **通知状态更新接口废弃**:通知已读标记接口在代码中标注为已废弃,假定其功能仍可用但可能在后续版本移除。
