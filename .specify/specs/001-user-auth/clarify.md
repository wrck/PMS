## AMB-001-01: `user` 与 `fnd_user_info` 双用户表并存
- **决策结论**: 采纳解释 1——`user` 为 `fnd_user_info` 的同义视图/别名,指向同一物理表。新系统统一为单一用户表(`fnd_user_info`),迁移期若需兼容则用视图别名过渡。
- **依据**: 日志查询、委派查询等只读场景与主流程写操作若指向不同物理表,会导致密码更新后日志仍查老数据,生产环境不可接受;两表字段大体一致,混用属历史命名习惯。
- **影响范围**: FR-010(密码重置)、FR-022(日志查询补全 realName)、FR-024(委派查询 owner/assignee);数据契约 user 表分级;SC-007(日志审计真实姓名补全)。
- **回滚提示**: 撤销决策时需将 user 表重新视为独立物理表,排查双写一致性并补齐迁移脚本;新系统需恢复双表兼容查询逻辑。

## AMB-001-02: `role` 与 `fnd_roles` 双角色表并存
- **决策结论**: 采纳解释 1——`role` 为 `fnd_roles` 的别名/视图,与 AMB-001-01 保持一致策略。新系统统一为 `fnd_roles`,迁移期用视图兼容。
- **依据**: 若展示用角色名与实际授权角色来自不同表,用户看到的角色名可能与实际权限不符,存在安全隐患;两表字段(id/roleName/status)对应一致。
- **影响范围**: FR-005(用户列表角色名展示)、FR-007(编辑用户角色关联);数据契约 role 表分级;SC-003(权限校验中角色判定)。
- **回滚提示**: 撤销决策时需将 role 表视为独立物理表,排查角色名展示与授权数据一致性,恢复双表查询逻辑。

## AMB-001-03: 区域权限市场/用服映射规则启用状态不明
- **决策结论**: [暂定决策] 倾向解释 2(已禁用)——新系统不实现 16x↔31x 前缀互换规则,仅保留 dpNo 合并逻辑;若业务反馈区域权限缺失再补回。需查 `UserUtil.processAreaPower` 源码确认,建议业务确认。
- **依据**: 源码中 `LoginServiceImpl.processAreaPower` 内联逻辑已注释,注释通常表示废弃意图;新系统不应继承未确认的隐式规则。
- **影响范围**: FR-001(登录区域权限计算)、SC-014(区域权限合并验收);用户登录后可见的数据范围(areapower 决定部门级数据权限)。
- **回滚提示**: 撤销决策时需重新实现 16x↔31x 前缀互换规则,并在登录流程中启用;需先确认 UserUtil.processAreaPower 的实际行为。

## AMB-001-04: 用户角色变更联动(changeType)规则与失败处理缺失
- **决策结论**: [暂定决策] 采纳解释 1(失败不回滚,仅记录日志)——changeType=service 更新该用户任服务经理的所有项目(服务经理字段替换为 newMemberCode);program 同理对项目经理;both 两者都更新。联动失败时不回滚用户编辑,仅记录日志。新系统建议改为异步消息(解释 3)更健壮。需确认 changeType 三值与项目字段的精确映射,建议业务确认。
- **依据**: 联动为外部项目工具调用(`ProjectUtils.updateServiceAndProgramMember`),跨系统事务成本高;用户编辑是本域核心职责,不应被外部系统故障阻断。
- **影响范围**: FR-007(编辑用户)、SC-013(角色变更联动验收);用户离职/转岗后项目归属正确性;跨域依赖(项目域)。
- **回滚提示**: 撤销决策时需改为事务回滚或异步消息模式;需补充 changeType 三值与项目字段的精确映射规则并实现补偿机制。

## AMB-001-05: 角色默认页面硬编码 `module/Welcome1.action`
- **决策结论**: 采纳解释 2(改为可配置)——新系统将 defaultPage 作为角色表单的可选输入,移除硬编码,留空时回退至系统默认页。
- **依据**: 代码注释(RoleManageAction.java)已表明开发意图为去硬编码("最好不要写死,后期需要调整");角色级默认页在多角色场景下可作为角色切换的落地页。
- **影响范围**: FR-013(新增角色)、FR-014(编辑角色)、数据契约 fnd_roles.defaultPage 字段;US-12/US-13(角色管理验收)。
- **回滚提示**: 撤销决策时需恢复 defaultPage 硬编码为 `module/Welcome1.action`,移除表单输入项;或确认角色级 defaultPage 未被使用后标记为 D 级字段。

## AMB-001-06: `menuCode` 唯一性约束未确认
- **决策结论**: 采纳解释 1(有唯一约束,DDL 层)——menuCode 在数据库层面有唯一索引。新系统应在 menuCode 上显式建唯一索引;若 DDL 确认无约束,则需排查是否存在重复数据并补建。
- **依据**: menuCode 作为跨表关联键(fnd_user_menus.menuCode → fnd_menus.menuCode),若无唯一性保证,用户菜单授权查询将产生一对多歧义,生产系统不可接受。
- **影响范围**: 数据契约 fnd_menus.menuCode、fnd_user_menus.menuCode;FR-015(菜单管理)、FR-001 步骤5(登录加载用户菜单权限);菜单授权一致性。
- **回滚提示**: 撤销决策时需移除 menuCode 唯一索引,改为应用层保证不重复,并排查历史重复数据。

## AMB-001-07: 测试环境免密登录语义歧义
- **决策结论**: 采纳解释 1(用户名必须存在有效,密码任意)——测试环境下用户名必须存在且 status=1,系统跳过密码比对,密码任意(或为空)即可登录。建议确认 `sys.envirment.argu` 的完整值域与非生产值的行为。
- **依据**: FR-001 步骤2 仍执行"按用户名查询有效用户",步骤3 仅跳过步骤4 的密码比对;"使用库中密码"语义为系统内部替换而非用户提交;完全免登录(解释2)存在安全隐患。
- **影响范围**: FR-001(账号密码登录)、US-001 场景5、Edge Case"用户不存在";测试环境安全策略;SC-004(密码安全验收,测试环境豁免范围)。
- **回滚提示**: 撤销决策时需重新定义测试环境登录语义(如改为完全免登录或要求密码比对),并调整 sys.envirment.argu 值域处理逻辑。

## AMB-001-08: CAS 服务器地址硬编码与可开关配置矛盾
- **决策结论**: 采纳解释 1(参数化)——新增系统参数 `sys.cas.server.url`,FR-002/FR-004 从参数读取 CAS 服务器地址,支持多环境部署。
- **依据**: NFR-010 已确立 CAS 可开关的设计意图,地址参数化是同一配置化思路的延伸;多环境(开发/测试/生产)CAS 服务器地址不同,硬编码将阻碍部署。
- **影响范围**: FR-002(CAS 登录)、FR-003(CAS 单点登出)、FR-004(主动登出);NFR-010(CAS 可开关);部署与环境配置策略。
- **回滚提示**: 撤销决策时需恢复 CAS 地址硬编码 `https://cas.dptech.com:8443/`,移除 sys.cas.server.url 参数;或改为通过外部 properties 文件管理。

## AMB-001-09: `menuPower` 权限编码语义反直觉

## AMB-001-10: 老架构/新架构行为并存于同一 FR,契约边界模糊
- **决策结论**: 采纳解释 2(两套均为契约,新系统同时支持)——用户菜单授权(绕过角色的直接授权)与用户角色关联(通过角色的间接授权)是两种互补的权限模型,新系统保留双模型,但统一端点为 REST 风格。
- **依据**: FR-001 登录时同时加载用户菜单权限(menuCode→menuValue)和角色菜单权限,说明两者并存是设计而非遗留;两种授权模型业务上均有需求(直接授权用于特殊用户,角色授权用于批量管理)。
- **影响范围**: FR-006~FR-009(用户 CRUD)、FR-016/FR-017(授权绑定);数据契约(用户菜单授权 vs 用户角色关联两套模型);新系统授权模型设计。
- **回滚提示**: 撤销决策时需选择单一授权模型(仅角色关联或仅直接授权),并迁移另一种模型的数据;需重新评估登录权限加载流程。

## AMB-001-11: 邮件发送失败重试上限与补偿机制缺失
- **决策结论**: 采纳解释 2(固定次数 + 失败标记)——新系统扩展 sendFlag 值域(0=未发、1=已发、2=失败),达到重试上限后标记失败并告警。建议确认老代码是否有重试计数字段(如 retryCount)或仅靠 sendFlag 判断。
- **依据**: 无限重试(解释1)在邮件服务器持续不可用时会积压大量请求;固定次数 + 失败标记可避免积压并提供人工介入入口。
- **影响范围**: SC-009(邮件可靠性验收)、US-17(邮件通知);fnd_mails.sendFlag 值域(需扩展失败态);账号开通/密码重置通知的可到达性。
- **回滚提示**: 撤销决策时需恢复无限重试策略(sendFlag 始终为 0 直到成功),移除失败态标记;或改为指数退避重试。

## AMB-001-12: 多租户 `compId` 隔离在老架构缺失导致双语义
- **决策结论**: 采纳解释 1(UserRole 表 + compId)——新系统采用 UserRole 关联表(userId+roleId+compId)表达多租户角色关联;roleIds 串仅作迁移期只读兼容,迁移完成后标记为 D 级。需确认 UserInfo 扩展信息表的字段结构(spec 中未给出完整 DDL)。
- **依据**: NFR-007 已明确多租户隔离为新架构能力;roleIds 串无法表达"同一用户在不同公司有不同角色"的多租户语义;UserRole 表是多租户角色关联的标准模型。
- **影响范围**: FR-006~FR-009(用户 CRUD)、FR-016/FR-017(角色绑定);数据契约 fnd_user_info.roleIds vs UserRole 表;NFR-007(多租户隔离);SC-005(跨租户越权拦截)。
- **回滚提示**: 撤销决策时需保留 roleIds 串模型并引入 compId 字段到用户主表,或采用双模型并存;需重新设计多租户角色查询逻辑。
