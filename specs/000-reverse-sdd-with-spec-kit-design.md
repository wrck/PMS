# 逆向 Spec-Driven Development 方法论设计

- **日期**: 2026-07-09
- **主题**: 用 Spec Kit 对既有 Java 系统做逆向工程,输出可用于新系统(AI 全新生成实现、复用/迁移数据库)开发阶段的全部文档
- **产出范围**: `constitution` + `spec` + `clarify`(行为契约固化层,后续新功能/重构需求可在此之上接 `/plan` `/tasks`)
- **状态**: 待用户 review

---

## 1. 背景与目标

### 1.1 现状
- 老系统:完整 Java 源码 + 零散且过时的文档,文档存在理解歧义。
- 文档与代码长期脱节,知识碎片化,新成员上手成本高,重构风险大。

### 1.2 新系统目标
- 同 Java 语言,但更换技术栈(框架/ORM 等)。
- 实现由 AI Agent 全新生成。
- 数据库**最大程度复用**,或以最简便方式迁移历史数据。

### 1.3 本方法论要解决的问题
Spec Kit 原生面向**正向设计**(需求 → spec → plan → 代码)。逆向工程需要"倒着走":从既有代码反推规格,再正向固化成 Spec Kit 标准 artifact。本设计定义这条反向抽取 + 正向固化的双轨流程,输出经澄清、无歧义、技术栈无关的行为契约,作为新系统开发的唯一事实来源。

### 1.4 产出边界
- **产出**:constitution、spec、clarify 三类 artifact(按业务域分域产出)。
- **不产出**:plan、tasks、implement(留给后续新功能/重构需求触发)。
- **停止条件**:所有 spec 内 `[待澄清]` 标记清零,数据契约分级完成。

---

## 2. 核心思路:反向抽取 + 正向固化

双轨并行:

- **反向抽取**:由 AI Agent(在 TRAE 中即本 Agent)从老 Java 源码 + 零散文档反推 spec/constitution **草稿**。草稿带代码证据,不确定处标记 `[待澄清]`。
- **正向固化**:把草稿喂给 Spec Kit 的 `/speckit-constitution` → `/speckit-specify` → `/speckit-clarify` 命令链,做结构化、去歧义、落盘成标准 artifact。

关键约束贯穿全程:
- **SPEC-TYPE-01**:spec 必须技术栈无关,禁止绑定 Spring/JPA 等具体框架注解或类名。
- **DATA-REUSE-01**:数据库表结构视为契约,新系统默认复用,变更需显式 spec。
- **AMBIGUITY-01**:所有歧义点必须在 `/clarify` 阶段记录决策与依据,spec 内不得残留 `[待澄清]`。

---

## 3. 阶段 0:环境与仓库准备

### 步骤
1. 安装 specify CLI:
   ```
   pipx install git+https://github.com/github/spec-kit.git
   ```
2. 在老系统代码仓库(或其副本)初始化:
   ```
   specify init . --ai <你的agent>
   ```
3. 为每个业务域建 feature branch:`001-<domain-name>`(Spec Kit 靠 git 分支隔离 feature,后续逐域切换分支产出 spec)。
4. 在 TRAE 中打开该仓库,后续所有 `/speckit-*` 命令由本 Agent 执行。

### 约束
- spec 技术栈无关:不写 Spring/具体 ORM 细节,只描述行为契约。技术决策推迟到后续 `/plan`。
- 每个业务域独立 branch、独立 spec,互不阻塞。

---

## 4. 阶段 1:Constitution 反推(项目宪法)

### 目标
从代码反推架构风格、技术约束、编码规范、数据复用优先原则,固化成 constitution artifact。

### Spec Kit 映射
`/speckit-constitution`

### 步骤
1. AI 扫描:`pom.xml`/`build.gradle`、目录结构、基类、注解、异常处理、日志、配置。
2. 抽取:分层架构、框架版本、命名约定、事务策略、安全机制、DB 访问方式。
3. 注入三条逆向特有原则(SPEC-TYPE-01 / DATA-REUSE-01 / AMBIGUITY-01)。
4. 正向固化:把抽取结果作为参数喂给 `/speckit-constitution`。

### 提示词 1-A:反向抽取 constitution 草稿(给本 Agent)
```
你是逆向工程架构师。扫描当前 Java 仓库,反推项目宪法。输出分 7 节:
1. 架构风格(分层?DDD?微服务?事件驱动?)
2. 技术栈与版本(从构建文件提取,仅记录现状,不作新系统约束)
3. 分层与模块边界(每层职责、依赖方向)
4. 编码规范(命名、包结构、注解约定、异常体系)
5. 横切关注点(事务、日志、安全、缓存策略)
6. 数据访问方式(JPA?MyBatis?JDBC?分库分表?)
7. 现存技术债与风险(为后续重构标靶)

每节附 3-5 条代码证据(文件:行号)。不要猜测,无证据则标注 [待澄清]。
```

### 提示词 1-B:正向固化
```
/speckit-constitution 基于 <贴入1-A输出> 固化项目宪法。
追加三条强制原则:
- SPEC-TYPE-01: spec 必须技术栈无关,禁止绑定具体框架注解/类名
- DATA-REUSE-01: 数据库表结构视为契约,新系统默认复用,变更需显式 spec
- AMBIGUITY-01: 所有歧义点必须在 /clarify 阶段记录决策与依据
```

---

## 5. 阶段 2:Spec 反推(核心,分两步)

### 2a. 业务域切分(DDD 战略)

#### 目标
把老系统切成可独立规格化的 bounded context,每域一个 spec feature branch。

#### 提示词 2-A:域切分
```
你是 DDD 战略分析师。基于阶段1的模块边界 + 包结构 + 数据库表归属,把系统切分为
bounded context。每个域输出:
- 域名(中英文)
- 一句话职责
- 核心聚合根(从实体类反推)
- 涉及的数据库表清单
- 与其他域的依赖关系(上下游)
- 建议的 spec feature branch 编号(001-xxx)

目标:每域可独立 /specify,互不阻塞。域数控制在 5-10 个,过大则再切。
```

切完后,为每个域 `git checkout -b 001-<domain>`,逐域执行 2b。

### 2b. 每域抽取 spec 四要素

#### Spec Kit 映射
`/speckit-specify`(逆向时输入是"反推结果"而非"需求描述")

#### 提示词 2-B:反向抽取域 spec 草稿(给本 Agent,逐域执行)
```
你是逆向规格分析师。针对 <域名> 域,扫描相关 controller/service/repository/entity,
反推规格,严格技术栈无关。输出 4 章:

第1章 用户故事:从 controller 端点反推 actor + 目标 + 价值(As a... I want... so that...)
  - 每个端点至少1个故事,标注 HTTP 方法+路径作为证据
第2章 功能需求:按用例组织,每条含 触发条件/输入/处理规则/输出/异常
  - 处理规则从 service 业务逻辑反推,禁止出现类名/注解
第3章 数据契约【最关键】:列出该域所有表
  - 表名、字段名、类型、是否可空、语义说明、业务不变量(唯一/外键/枚举值域)
  - 标注哪些字段是"对外契约字段"(新系统必须保留),哪些是"内部实现字段"(可重构)
  - 此章是新系统复用/迁移 DB 的唯一事实来源
第4章 非功能需求:从事务注解/缓存/限流配置反推 性能/一致性/可用性要求

每条需求附代码证据(文件:行号)。不确定处标 [待澄清]。
```

#### 提示词 2-C:正向固化
```
/speckit-specify 将 <贴入2-B输出> 结构化为 Spec Kit spec artifact。
要求:
- 描述"系统做什么"而非"代码怎么做"
- 第3章数据契约单独成节,用表格呈现,字段语义必须明确
- 保留所有 [待澄清] 标记,留给下一步 /clarify
```

---

## 6. 阶段 3:Clarify 澄清(消化零散文档歧义)

### 目标
把零散过时文档的歧义、代码与文档不符处,结构化为"问题 → 决策 → 依据"记录,清零所有 `[待澄清]`。

### Spec Kit 映射
`/speckit-clarify`

### 步骤
1. AI 比对:代码行为 vs 零散文档,找出矛盾/缺失/多解。
2. 对每个 `[待澄清]` 点生成结构化提问。
3. 逐条与用户确认,记录决策。

### 提示词 3-A:歧义识别
```
你是规格澄清师。针对 <域名> 的 spec 草稿,执行三比对:
1. 代码 vs 现有文档:列出所有矛盾点(文档说A,代码做B)
2. 代码 vs 代码:同名字段/枚举在不同处语义不一致的情况
3. spec 内部:[待澄清] 标记逐条展开

每条歧义输出:
- ID: AMB-<域>-<序号>
- 位置:涉及文件:行号 + 涉及文档片段
- 现象:矛盾/缺失/多解的具体描述
- 候选解释:2-3 种可能含义
- 影响面:波及哪些需求/数据字段
- 建议决策:基于代码证据的推荐选项+理由
```

### 提示词 3-B:正向固化
```
/speckit-clarify 聚焦 <域名> 的 AMB-* 歧义清单。
对每条:采纳建议决策(或用户指定),写入 clarify artifact,字段含
决策结论/依据/影响范围/回滚提示。
完成后 spec 内不得残留 [待澄清]。
```

## 7. 数据库复用/迁移专项(贯穿阶段 2-3)

数据契约在 spec 第 3 章固化,按下表分级处理:

| 维度 | 处理策略 |
|---|---|
| 表结构 | 默认全量复用,新系统 ORM 只做映射不改表 |
| 字段分级 | 契约字段(保留)/ 内部字段(可重构)/ 废弃字段(标注不迁移) |
| 枚举值域 | 固化为 spec 约束,新系统不得擅自扩值 |
| 历史数据迁移 | 若需迁移,在 spec 附"迁移不变量"(行数对账、外键完整性校验规则) |
| 分库分表 | 现有分片策略作为非功能需求记录,新系统须兼容 |

字段分级定义:
- **契约字段**:对外暴露或被其他系统/报表依赖,新系统必须保留语义与值域。
- **内部实现字段**:仅服务当前代码内部逻辑,可随重构调整。
- **废弃字段**:代码已不读写,迁移时标注不迁移。

## 8. 产出校验清单(阶段 3 完成后自查)

- [ ] 每个 spec 无残留 `[待澄清]`
- [ ] spec 不含 Spring/JPA 等框架专属名词(技术栈无关)
- [ ] 每张表都有字段语义说明 + 契约/内部分级
- [ ] 每条需求有代码证据可追溯
- [ ] constitution 含三条逆向原则(SPEC-TYPE-01 / DATA-REUSE-01 / AMBIGUITY-01)
- [ ] 域间依赖无环

## 9. 产出物清单

| Artifact | 数量 | 位置 | 说明 |
|---|---|---|---|
| constitution | 1 | 仓库根 `.spec/` 或 Spec Kit 默认路径 | 全局项目宪法 + 三条逆向原则 |

## 10. 后续衔接(超出本次产出范围,仅说明)

当新功能或重构需求到来时,在已有 constitution + spec + clarify 基础上:
1. 新建 feature branch,继承相关域 spec。
2. 在 spec 中追加/修改需求(保持技术栈无关)。
3. 视需要走 `/clarify` → `/plan`(此时选定新技术栈)→ `/tasks` → `/analyze` → `/implement`。

新系统的技术选型决策集中在 `/plan` 阶段一次性做出,与行为契约解耦。

---

## 11. 落地路径

本方法论可直接在 TRAE 中执行,有两种走法:

1. **方法论交付(已完成)**:本文档即操作手册,团队可按阶段 0-3 自行执行。
2. **直接落地到老系统**:提供老 Java 仓库路径(或把代码放进 `/workspace`),由本 Agent 从阶段 0 开始,逐域执行反推 + 固化,产出真实 artifact。
