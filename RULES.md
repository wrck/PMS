# 工作规则（硬性强制）

> 本规则适用于所有在主仓库 `/workspace` 及其子目录工作的代理（主代理与子代理）。
> 任何违反本规则导致的代码丢失、提交错乱、数据不一致，均视为重大事故。
> 本规则与 `AGENTS.md` 并列，共同构成上下文约束。

---

### 规则 1：子代理工作树必须合并回主仓库并推送远端

当任何子代理（通过 Task 工具派发）完成其独立工作树内的所有任务和修改后，**必须立即通过标准提交流程将变更合并到主仓库并推送到远端**，确保代码库的完整性和一致性以及远端备份。

- 子代理不得仅在自身临时工作树内提交后即返回，必须确认提交已落到主仓库。
- 子代理在主仓库完成 `git commit` 后，**必须立即执行 `git push` 将提交推送到远端对应分支**，不得仅停留在本地。
- 合并前必须执行本地测试（编译、类型检查），确保没有引入新的错误或冲突。
- 主代理在派发子代理任务时，**必须在任务描述中显式要求子代理将最终提交同步到主仓库路径并推送到远端**，而非子代理的临时路径。
- 推送失败时必须立即排查并重试，确认远端 `git log` 中已出现该提交后方可视为任务完成。

### 规则 3：每次提交后必须立即推送远端

**任何代理（主代理或子代理）在主仓库内完成 `git commit` 后，必须立即执行 `git push` 将该提交推送到远端对应分支**，不得积压推送。

- 推送是提交流程的强制组成部分，未推送的本地提交等同于未完成。
- 主代理在自行提交后同样适用本规则。
- 推送失败的提交必须立即修复并重试，禁止保留未推送的本地提交堆叠。
- 仅当远端 `git log` 中确认出现提交哈希后，方可视为该提交真正落地。

### 规则 2：验证通过后立即提交

任何任务在经过验证确认正确完成后，**必须立即进行提交操作**，不得积压。

- 提交信息必须清晰、规范，包含：任务内容、实现方式、相关测试结果。
- 任务完成标准：功能实现符合需求规格 + 通过所有相关测试用例 + 代码质量符合项目标准。
- 禁止"先做完所有任务再一起提交"的积压模式——每完成一个可验证的功能单元即提交一次。

---

## 附则

- 本规则优先级高于 `AGENTS.md` 中的任何冲突条款。
- 本规则随项目演进持续更新，每次更新需在提交信息中注明规则变更内容。
- 所有代理（主代理与子代理）在开始工作前必须读取并遵守本规则。
